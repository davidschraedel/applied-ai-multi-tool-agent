Retrieval-Augmented Generation (RAG)
=====================================

RAG is a technique that improves LLM responses by retrieving relevant external documents before generating a response. Instead of relying solely on the LLM's training data, RAG adds a retrieval step that fetches up-to-date or domain-specific context.

How RAG Works
--------------
1. Indexing: Documents are split into chunks, converted to vector embeddings, and stored in a vector store.
2. Retrieval: When a user asks a question, the query is embedded and the most semantically similar chunks are retrieved.
3. Generation: The retrieved chunks are included in the prompt as context, and the LLM generates an answer grounded in that context.

Key Components
---------------
- **Text Splitter**: Breaks documents into manageable chunks (e.g., 500 characters with 50-character overlap).
  - Class: RecursiveCharacterTextSplitter from @langchain/textsplitters
- **Embeddings Model**: Converts text to dense vector representations.
  - For this project: OpenAIEmbeddings with text-embedding-3-small (requires OPENAI_API_KEY)
  - Note: Anthropic does not offer embeddings. OpenAI or Voyage AI must be used even when Claude is the chat model.
- **Vector Store**: Stores and retrieves embeddings. For in-memory (demo scale): MemoryVectorStore from @langchain/classic/vectorstores/memory
- **Similarity Search**: Returns top-k chunks most similar to the query.
  - Method: vectorStore.similaritySearch(query, k)

Source Attribution
-------------------
Each retrieved document has a metadata.source field containing the file path. Always include this in the tool's return string so the user knows where the information came from.

Example return format:
  [Source: src/docs/langchain-overview.txt]
  LangChain is an open-source framework...

Chunk Size Tuning
------------------
- Smaller chunks (200-300 chars): More precise retrieval, but may lose context
- Larger chunks (500-1000 chars): More context, but may include irrelevant text
- Start with 500 chars / 50 overlap and adjust based on retrieval quality
