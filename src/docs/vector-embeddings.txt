Vector Embeddings and Semantic Search
=======================================

Vector embeddings are numerical representations of text that capture semantic meaning. Two pieces of text that mean the same thing (even if worded differently) will have embeddings that are mathematically close to each other. This property enables semantic search — finding relevant content by meaning rather than exact keyword matching.

How Embeddings Work
--------------------
1. A text embedding model (e.g., OpenAI's text-embedding-3-small) converts a string into a high-dimensional vector (e.g., 1536 dimensions for text-embedding-3-small).
2. These vectors are stored in a vector database or in-memory store.
3. At query time, the query is also embedded.
4. The similarity between the query vector and all stored document vectors is computed (usually using cosine similarity).
5. The top-k most similar document chunks are returned.

OpenAI Embedding Models
------------------------
- text-embedding-3-small: 1536 dimensions, fast and cheap, recommended for most use cases
- text-embedding-3-large: 3072 dimensions, higher quality, more expensive
- text-embedding-ada-002: Legacy model, still widely used

For this project, text-embedding-3-small is used. It requires an OPENAI_API_KEY even when Claude is the chat LLM.

MemoryVectorStore
------------------
For development and demos, LangChain's MemoryVectorStore (from @langchain/classic/vectorstores/memory) provides an in-memory vector store with no external dependencies. Documents are embedded on startup and stored in RAM. The store is lost when the process restarts, which is acceptable for demo purposes.

Production alternatives: Pinecone, Weaviate, Chroma, pgvector (PostgreSQL extension).

Similarity Search
------------------
The similaritySearch(query, k) method returns the top k document chunks most similar to the query. Each result is a Document object with:
- pageContent: The text content of the chunk
- metadata.source: The source file path (crucial for attribution)

Chunking Strategy
------------------
Large documents must be split into smaller chunks before embedding because:
1. Embedding models have token limits
2. Smaller chunks lead to more precise retrieval

RecursiveCharacterTextSplitter splits on paragraph breaks, then sentence breaks, then word breaks — preserving as much semantic coherence as possible.

Recommended starting parameters:
  chunkSize: 500    (characters)
  chunkOverlap: 50  (characters shared between adjacent chunks)
